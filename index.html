1 Scope

This document specifies a normative JSON syntax binding for the OASIS Universal Business Language (UBL). It defines the representational rules, structural constraints, and validation requirements by which UBL business documents and their constituent components are to be expressed and verified in JSON [RFC8259].

The specification applies to complete UBL documents [UBL] as well as to individual Aggregate Business Information Entities (ABIEs) when exchanged as autonomous JSON payloads. It does not modify the UBL Semantic Library or the catalogue of standardized business documents but only prescribes the binding of those artefacts to JSON.

3 Document Conventions

3.1 Key Words

Within the normative text of this specification, the terms "shall", "shall not", "should", "should not" and "may" are to be interpreted as described in Annex H of ISO/IEC Directives, Part 2, Principles and rules for the structure and drafting of ISO and IEC documents [ISO/IEC Directives].

3.2 Typographical Conventions

  • Naming conventions:

    • Names of Business Information Entities (BIEs), including Document ABIEs, ABIEs, BBIEs, and ASBIEs, appear in Title Case (e.g., Invoice, Order, ID, IssueDate, BuyerCustomerParty).

    • CCTS-derived supplementary components (e.g., members of typed primitives such as Identifier, Code, Amount, Quantity, Measure, BinaryObject, Date, Time, Text/Name) appear in lowerCamelCase (e.g., schemeID, currencyID, unitCode).

  • Monospaced font is used for JSON keywords, property names, schema keywords, literal values, and regular expressions (e.g., $schema, currencyID, true, ^[0-9]{4}-[0-9]{2}-[0-9]{2}$).

4 Introduction

Universal Business Language (UBL) is an international standard published as ISO/IEC 19845. It consists of a semantic library of reusable business information entities and a catalogue of standardized business documents.

This document defines the JSON syntax binding for UBL 2.5. It prescribes the normative conventions by which UBL business documents and Aggregate Business Information Entities (ABIEs) are represented in JSON, ensuring consistency with the UBL Semantic Library and interoperability across syntax bindings.

The specification extends conformance to both complete documents and individual ABIEs, permitting their use as standalone JSON payloads as well as embedded components within documents.

This document does not specify API operations, transport protocols, security frameworks, or workflow processes. Its scope is limited to the binding of UBL to JSON.

5 Use of UBL JSON

5.1 Exchange of complete UBL documents

The predominant use of UBL JSON is the exchange of complete business documents such as Invoice, Order, Catalogue, or BillOfLading. In this mode, the root object of the JSON instance corresponds directly to a document type defined in the UBL Semantic Library. Each such instance shall conform to the normative rules of this specification, including property naming, cardinality, datatype representation, and schema identification via the $schema property as specified in section 9.1.

This document-oriented approach reflects the original design objective of UBL: to model electronic transactions that mirror existing business practices. Documents are the units of contractual and legal effect in commerce, and UBL JSON preserves that role while allowing instances to be expressed in JSON syntax. For example, an electronic Invoice sent from a supplier to a buyer would appear as a JSON object rooted in the Invoice ABIE. It may contain embedded Party objects (AccountingSupplierParty, AccountingCustomerParty), LineItem arrays, and PaymentMeans details, all governed by the same semantic library as in UBL XML.

Document-level UBL JSON instances are therefore suited for use in environments where a complete business transaction is transmitted between systems or organizations.

5.2 Use of ABIEs as standalone payloads

This specification also defines the use of individual Aggregate Business Information Entities (ABIEs) as valid standalone JSON payloads. In this mode, an ABIE functions as the root of the JSON instance and is treated as a self-contained resource.

This design enables UBL to be employed in API-driven integrations. For example:

  • A Party ABIE may be exposed through endpoints such as GET /party/{id} or PUT /party/{id}.

  • An Address ABIE may be returned in response to a geolocation lookup.

  • A PaymentMeans ABIE may be submitted to configure a customer’s payment preferences.

The semantics remain consistent: the Party appearing inside an Invoice is structurally identical to the Party served at an API endpoint. This eliminates semantic drift between documents and APIs, ensuring that business data carries the same meaning across channels.

Conformance rules are identical to those for full documents: property naming, cardinality, datatype representation, and schema identification through $schema. Communities of use may therefore adopt either or both paradigms, knowing they are interoperable by design.

5.3 Complementarity of documents and components

The dual applicability of UBL JSON, as both document-level exchange and ABIE-level payloads, accommodates a broad spectrum of integration requirements.

  • Document-centric exchanges continue to serve their traditional role in legally binding and auditable transactions.

  • Component-centric exchanges enable fine-grained data synchronization, lightweight APIs, and modular system integration.

Together, they provide a single, coherent semantic model that spans from high-level contract formation to low-level resource management. This flexibility allows organizations to modernize integration strategies incrementally, without abandoning the established document workflows that underpin global commerce.

5.4 Non-goals

While UBL JSON instances may be carried over web APIs, message queues, file transfers, or other channels, this document does not define:

  • API operations, endpoints, or protocol bindings;

  • transport, messaging, or packaging formats;

  • authentication, authorization, or encryption models;

  • lifecycle management or version negotiation workflows.

Such matters are left to communities of use, interoperability frameworks, or implementation profiles. The role of this specification is limited to prescribing the normative rules by which UBL business documents and components are represented in JSON, ensuring that a valid instance has the same meaning wherever it is used.

6 Document model

6.1 Aggregate structures and namespacing

Each Aggregate Business Information Entity (ABIE) shall be represented as a JSON object, the constituent properties of which are the Business Information Entities (BIEs) defined in the UBL Semantic Library [UBL]. Property names shall be taken verbatim from the UBL Semantic Library, case-sensitive, and governed by the UBL Naming and Design Rules [UBLNDR], whereby BIEs appear in Title Case and CCTS-derived supplementary [CCTS] components appear in lowerCamelCase.

No additional properties shall be permitted unless explicitly provided for in the definition of the object type or introduced through the extension mechanism described in section 6.3. Any instance of a typed BBIE or ABIE that contains members beyond those prescribed by the schema or the defined extension container shall be deemed invalid.

6.2 References between entities

Each Basic Business Information Entity (BBIE) is typed by a UBL data type derived from the Core Component Technical Specification (CCTS) [CCTS]. Accordingly, every BBIE shall be represented in JSON using the scalar form or object form as prescribed in section 7.

An Associated Business Information Entity (ASBIE) that references another ABIE shall embed that ABIE in full, either as a nested object (for singleton occurrences) or as an array of such objects (for repeating cardinalities). Indirection by means of identifiers alone is not defined by this specification. ASBIEs shall always contain the full JSON representation of the referenced ABIE, conforming to the rules applicable to that ABIE. For repeating BIEs, the provisions of section 7.1 regarding cardinality and repetition shall apply.

6.3 Document extensions

UBL provides a standardized extension scaffolding defined in the Common Extension Components schema (see 12.4.1.5). This scaffolding enables user communities to introduce additional content into a UBL JSON instance when required.

An extension container, if present, shall be expressed using the UBLExtensions property. Each extension shall be a JSON object containing at least the following members:

  • ExtensionURI: a URI that uniquely identifies the extension. This value shall not collide with any predefined UBL property name. Extension identifiers should be namespaced, for example by reverse-DNS or URI-scoped conventions, to minimize the risk of collision across communities.

  • ExtensionContent: a JSON object containing the extension payload. This member is mandatory whenever the extension container is used and shall not be empty.

Extension content shall not alter or redefine the meaning of standard UBL properties and is strictly additive in nature. Extension metadata (e.g., ExtensionAgencyID, ExtensionVersionID) is optional unless required by a community profile.

Implementers should be aware that extensions inevitably reduce interoperability with communities or systems that have not adopted the same extension.

Example (non-normative, truncated):

"UBLExtensions": [
    {
        "ExtensionURI": "https://example.org/custom/MyUBLExtension",
        "ExtensionContent": {"CustomProperty": "CustomValue"}
    }
]

7 Rules of representation

7.1 Cardinality and repetition

The cardinality of Business Information Entities (BIEs) as defined in the UBL Semantic Library shall be reflected in JSON according to the following rules:

  • [1..1]: The property shall be present exactly once, represented either in scalar or object form as prescribed for its data type.

  • [0..1]: The property may be absent, or, if present, shall appear exactly once in scalar or object form as prescribed for its data type. Omission shall constitute non-occurrence.

  • [0..n] or [1..n]:

    • If exactly one occurrence is present, it shall be represented as a single value or object.

    • If more than one occurrence is present, it shall be represented as an array of values or objects.

Consumers shall implement acceptance of both the singleton and array forms in order to accommodate the full range of cardinalities. Producers shall emit the scalar form only when no supplementary components are supplied or required and shall emit the object form whenever any supplementary component is present or mandated by a profile. The JSON literal null shall not be employed to indicate absence. Omission of an optional property shall constitute the normative representation of non-occurrence.

Example (non-normative). A BBIE Description with cardinality 0..n may be represented as:

"Description": "My description"

when a single occurrence is present, or as:

"Description": {"value": "My description", "languageID": "EN"}

When a single occurrence is present with a supplementary component, or as:

"Description": [
    {"value": "My description", "languageID": "EN"},
    {"value": "Mi descripción", "languageID": "ES"}
]

when multiple occurrences are present.

7.2 Nullability and omission

JSON null values shall not be employed to represent absence in UBL JSON. Optional properties should be omitted when not present. Where the schema provides variants denoted by the suffix _n, such variants indicate repeatability (0..n or 1..n) and, in limited cases, alternative representations (for example singleton versus array, or an object form for text values). The suffix _n is a schema-level construct and does not appear in instance JSON but serves solely to indicate multiplicity. These variants do not imply acceptance of JSON null.

7.3 Ordering, whitespace, and formatting

The order of members within a JSON object shall have no significance, and consumers shall not rely on property order. Producers may omit insignificant whitespace, and consumers shall disregard such whitespace in processing.

7.4 Error handling and strictness

Validation shall fail when any of the following conditions occur:

  • A JSON object contains members that are not permitted by the Semantic Library schema or by a declared extension.

  • A required property is absent.

  • A property is encoded in scalar form when supplementary components are present or required.

  • A property cardinality is violated, including cases where an array has fewer or more elements than permitted.

  • A scalar value does not conform to its declared lexical or pattern restrictions, for example a date or time not matching the prescribed regular expression.

  • A property is present with an empty string as its value.

  • An object defined by the Semantic Library is present but empty (i.e., contains no members).

  • In the case of natural language Text or Name elements with repeating cardinality:

    • Any occurrence omits a languageID, or

    • Two occurrences share the same languageID.

These rules apply in addition to JSON Schema validation and to any supplementary constraints introduced by an interoperability profile.

8 Controlled vocabularies

When a BBIE incorporates a schemeID, listID, unitCode, or currencyID, producers should employ published and stable vocabularies. Specifically, currencyID shall follow ISO 4217 [ISO 4217], unitCode should follow UN/ECE Recommendations 20 and 21 [UNECE Rec 20/21], and country or region codes shall follow ISO 3166 [ISO 3166]. Other code values should conform to the authoritative lists established by the relevant community of use. Conformance to such external vocabularies is a requirement of the profile or community in which the document is exchanged. In the absence of such a profile, validation of the values themselves lies outside the scope of this specification.

9 Versioning and profiles

9.1 Schema identification

Every conformant UBL JSON instance shall carry a $schema property at its root identifying the governing schema. For document instances, this value shall be the identifier of the corresponding UBL document schema. For standalone Aggregate Business Information Entity (ABIE) payloads, the value shall be the identifier of the Common Aggregate Components schema with the appropriate fragment identifier.

Out-of-band declarations of schema context (for example through API contracts or community profiles) may supplement this mechanism, but they do not replace the in-band $schema property. Where both are present, they shall be consistent.

For document instances, the root object shall include a $schema property whose value identifies the corresponding UBL document schema. The URI shall resolve to the authoritative schema published for that document type in the UBL JSON distribution.

For standalone Aggregate Business Information Entities (ABIEs) exchanged as autonomous root payloads, the instance shall include a $schema property whose value identifies the Common Aggregate Components schema together with a fragment identifier corresponding to the ABIE. Each ABIE in the UBL Common Aggregate Components schema is published with an anchor equal to its UBL name, enabling unambiguous reference.

For extensions expressed using the UBL extension mechanism (6.3), each extension container shall include a $schema property whose value identifies the schema context and version governing the extension content. The value shall be a stable identifier published by the authority responsible for the extension.

In all cases, producers shall ensure that instances validate successfully against the schema identified in the $schema property, and consumers shall apply the corresponding rule set. Where both in-band and out-of-band indications are provided, the profile authority or community of use shall define precedence. In the absence of such a rule, the indications shall be consistent.

9.2 Interoperability profiles

Communities of use may define profiles that constrain this base specification, for example by fixing codelists, tightening cardinalities, or introducing additional business rules. Such profiles shall not contradict or relax the normative structures defined herein. The mechanism for declaring profile conformance shall be specified by the profile authority. This specification requires only that such a declaration be explicit, unambiguous, and machine processable.

10 Data types

10.1 UBL unqualified data types

The CCTS-derived data types employed in UBL are represented in JSON either in scalar form (core value only) or in object form (a value property with supplementary components). Producers may emit the scalar form when no supplementary components are supplied or required. Producers shall emit the object form whenever any supplementary component is present. Consumers shall implement acceptance of both forms. Optional supplementary components are represented by omission, and the JSON literal null shall not be employed.

11 Digital Signatures

11.1 Introduction

Certain business or regulatory contexts require UBL documents to be digitally signed. Examples include invoices submitted under Continuous Transaction Control (CTC) regimes, tenders, or other documents requiring proof of authenticity and integrity.

UBL JSON integrates with the IETF JOSE framework, in particular JSON Web Signature (JWS) [RFC7515], to provide a standardized means of digitally signing UBL JSON documents. JWS is the JSON-native analogue to XMLDSig and is broadly supported across software stacks.

Digital signatures in UBL JSON support the following properties:

  • Integrity: the document has not been modified since it was signed.

  • Authenticity: the identity of the signer is certified by a trusted authority.

  • Non-repudiation: the signer cannot later deny having signed the document.

  • Timestamping: when combined with trusted timestamp tokens, the signature proves the document existed before a specific time.

Signature validity in UBL JSON relies on deterministic canonicalization of the instance. All signatures are applied over the JSON Canonicalization Scheme (JCS) [RFC8785] serialization of the business document, ensuring that semantically equivalent JSON structures yield identical octet streams for signing and verification.

This section defines the normative mechanism by which digital signatures are incorporated into UBL JSON instances. It binds the JWS format [RFC7515] to UBL JSON, prescribing the placement of signatures, the scope of canonicalization, and the handling of enveloped and detached signatures.

Cryptographic algorithms, key formats, certificate policies, and trust anchors are outside the scope of this specification and shall be defined by community and implementation profiles.

11.2 Signature Profiles

11.2.1 General

UBL JSON supports two complementary signature profiles:

  • Enveloped signatures: one or more JWS signatures are embedded within the UBL JSON document under the standard extension mechanism. The signed document remains a valid UBL JSON instance, and the signature is transported together with it.

  • Detached signatures: the JWS signature is conveyed separately from the UBL JSON instance. The UBL document remains unsigned in its payload form, while an external JWS object (or container format such as JAdES or ASiC-J) references and signs the document.

Both profiles ensure that a conforming UBL JSON document remains syntactically valid regardless of whether signatures are present.

11.2.2 Enveloped Signature Structure

Enveloped signatures shall be carried within the standard UBL extension mechanism (see section 6.3). A digital signature extension shall be identified by the reserved URI:

"ExtensionURI":
"https://docs.oasis-open.org/ubl/json/jws/enveloped"

The ExtensionContent member of such an extension shall contain one or more JSON Web Signature (JWS) objects expressed in the JSON serialization defined by [RFC7515].

The payload of each JWS shall be the JSON Canonicalization Scheme (JCS) canonicalized octet stream of the enclosing UBL JSON instance, with the exception that the extension container holding the signature shall be omitted from the canonicalization input. This ensures that the signature covers the complete business content of the document while avoiding recursive inclusion of the signature itself.

Each JWS object may include protected headers in accordance with [RFC7515], such as alg to identify the algorithm, kid to reference the signing key, or x5c to carry the certificate chain. Where multiple signatures are applied to the same document, the ExtensionContent shall carry an array of JWS objects, each independently verifiable over the same canonicalized payload.

A conformant consumer shall validate an enveloped signature by canonicalizing the instance excluding the signature extension, verifying the JWS signature over that canonicalized octet stream, and applying any certificate or trust validation procedures mandated by the relevant community or regulatory profile.

Example (non-normative, truncated):

"UBLExtensions": [
  {
    "ExtensionURI": "https://docs.oasis-open.org/ubl/json/jws/enveloped",
    "ExtensionContent": {
      "signatures": [
        {
          "protected": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9",
          "signature": "Z3VhcmQtbWUtd2VsbC4uLg",
          "header": {"kid": "12345"}
        }
      ]
    }
  }
]

11.2.3 Detached Signatures

Detached signatures shall be conveyed separately from the UBL JSON instance. In this profile, the UBL document itself remains unsigned in its payload form, and the signature is transported as a distinct JWS object. The payload of the detached JWS shall be the JCS-canonicalized octet stream of the complete UBL JSON instance.

Where required, the UBL JSON instance may include a Signature Aggregate Business Information Entity (ABIE) referencing the detached signature by URI or identifier. In such cases, the SignatureMethod property of the ABIE shall be set to the fixed identifier:

"https://docs.oasis-open.org/ubl/json/jws/detached"

The mechanism by which detached signatures are packaged together with the UBL document, such as multipart envelopes, container formats, or service-specific transports, lies outside the scope of this specification, but any such mechanism shall ensure that the association between the instance and its detached signature is unambiguous and verifiable.

A conformant consumer shall validate a detached signature by canonicalizing the entire UBL JSON instance, verifying the JWS signature over that canonicalized octet stream, and applying certificate and trust validation in accordance with the governing profile.

11.3 Canonicalization

Prior to signing or verification, a UBL JSON instance shall be transformed into a deterministic octet stream using the JSON Canonicalization Scheme (JCS) [RFC8785].

The canonicalization input shall exclude the UBLExtensions element containing the signature itself. For detached signatures, the entire canonicalized root instance shall be covered.

Consumers shall reject any signature whose input does not match the JCS-canonicalized form of the unsigned instance.

12 JSON schemas

12.1 Normative status

The JSON Schemas [JSON-SCHEMA] distributed with this specification are normative artefacts. They prescribe the structural representation of the UBL Semantic Library and the catalogue of UBL business documents in JSON syntax. Conformance to this specification requires that instances validate successfully against the applicable JSON Schema.

These schemas govern the syntactic representation of UBL in JSON. Business-level constraints such as controlled vocabularies, profile specific rules, or regulatory mandates are addressed separately in sections 8 (Controlled vocabularies), 9 (Versioning and profiles), and 11 (Digital Signatures).

All normative schemas are published in the json/schemas/ directory of the release package. Each schema is identified by a stable HTTPS URL that is fixed at the major version level (e.g., https://docs.oasis-open.org/ubl/json/schemas/Invoice-2). Minor revisions do not change these identifiers.

12.2 Document instances

When a complete UBL business document is exchanged (such as an Invoice, an Order, or a Catalogue) the root JSON object shall identify its governing schema by means of the $schema property. The value of this property shall be the major-stable identifier corresponding to the UBL document type.

This requirement ensures that any receiving system can unambiguously determine the exact structural rules to apply when validating and processing the instance. Because the identifiers are stable across all minor revisions of the UBL 2.x line, a document referencing, for example, a UBL Invoice-2 remains valid regardless of whether the schema implementation in use is UBL 2.5, 2.6, or 2.x. This guarantees forward compatibility, thereby protecting investments in existing implementations and reducing the operational risk of upgrading to newer releases.

Document instances follow the canonical model prescribed by the UBL Semantic Library and thus embed complete sets of ABIEs and BBIEs in accordance with the schema definitions. By referencing the document schema explicitly, producers and consumers can rely on a shared contract: the producer asserts conformance to the UBL document model, and the consumer can validate, process, and archive the document with the assurance that its structure is compliant and interoperable.

12.3 Standalone ABIEs

The JSON schemas define each Aggregate Business Information Entity (ABIE) of the UBL Common Library not only as a component for embedding within larger structures but also as a valid root payload. This design enables implementers to employ individual ABIEs directly in transactional contexts that do not require the overhead of a complete UBL document instance. A common example is the deployment of application programming interfaces (APIs) that exchange business components in isolation, such as a getAddress or putParty endpoint.

In these scenarios, the $schema property serves to anchor the instance to the relevant definition in the Common Aggregate Components schema, and the fragment identifier designates the ABIE in question. The result is that a JSON instance representing, for example, an Address payload, carries the same normative semantics as when that same ABIE is embedded within a UBL document such as an Invoice or an Order.

This approach guarantees semantic consistency: a Party or Address exchanged as a standalone payload is precisely the same structure, subject to the same rules, as when appearing within a larger document. It also promotes reuse of the UBL semantic vocabulary across heterogeneous integration models, allowing implementers to unify document-centric exchanges and component-centric API calls under a single coherent set of rules.

A document issued by the party who acts as an agent for a transportation carrier or other agents to the party who gives instructions for the transportation services (shipper, consignor, etc.) stating the details of the transportation, charges, and terms and conditions under which the transportation service is provided. The document evidences a contract of carriage by sea and the acceptance of responsibility for the goods by the carrier, by which the carrier undertakes to deliver the goods against surrender of the document. A provision in the document that the goods are to be delivered to the order of a named person, or to order, or to bearer, constitutes such an undertaking.

13 Safety, Security, and Data Protection Considerations

13.1 Security considerations

UBL JSON instances are ordinary JSON documents. Implementers shall therefore consider the general security considerations applicable to application/json as described in [RFC8259], including validation of input, avoidance of unsafe evaluation of textual content, and proper handling of Unicode per [UTR#36].

The extension mechanism (section 6.3) allows communities to embed additional content. Such content is outside the scope of this specification and may introduce security risks if not governed by appropriate community rules. Implementers should ensure that extensions do not introduce executable code, unsafe references, or other payloads that could compromise consumers.

Where digital signatures are applied, the considerations described in [RFC7515] (JWS) and [RFC8785] (JCS) apply. In particular, weak algorithms, inadequate key sizes, and improper certificate validation may compromise integrity and authenticity. This specification deliberately leaves algorithm choice and trust management to implementation profiles or regulatory frameworks.

13.2 Data protection and privacy considerations

UBL JSON instances may contain personally identifiable information (PII) or commercially sensitive data, including names, addresses, tax identifiers, and financial amounts. This specification does not define mechanisms for redaction, encryption, or access control. Implementers and communities of use are responsible for ensuring that processing and exchange of UBL JSON documents comply with applicable data protection laws and contractual obligations.

14 Conformance

14.1 Core conformance

A UBL JSON instance is conformant to this specification when all the following conditions are satisfied:

  1. The instance validates against the applicable normative JSON Schema defined in section 12.4 (Normative schemas) and carries a $schema property identifying the governing schema. For document instances, this is the schema of the declared document type. For standalone ABIE payloads, this is the Common Aggregate Components schema with the designated fragment identifier;

  2. If present, extensions conform to the scaffolding defined in section 6.3 (Document extensions), do not collide with or alter the semantics of standard UBL properties, and extension metadata complies with the rules prescribed in that section; and

  3. The instance contains no content violations of the constraints listed in section 7 (Rules of representation).

14.2 Digital signature conformance

A UBL JSON instance that carries a digital signature is conformant to this specification only when the following conditions are satisfied:

  1. The signature is expressed as a JSON Web Signature (JWS) object conformant to [RFC7515] and bound to UBL JSON as prescribed in section 11 (Digital Signatures).

  2. When signatures are embedded in the instance, they are carried within the UBL extension mechanism (UBLExtensions) using the reserved extension URI https://docs.oasis-open.org/ubl/json/jws/enveloped.

  3. When signatures are conveyed separately from the instance, the payload of the detached JWS is the JCS-canonicalized octet stream of the complete UBL JSON document. Where the document includes a Signature ABIE referencing the detached signature, its SignatureMethod is set to https://docs.oasis-open.org/ubl/json/jws/detached.
